#데이터베이스 #동시성제어 #낙관적검증 #Validation #필수 #데이터베이스/동시성제어
## 정의
낙관적 동시성 제어, Optimistic Concurrency Control
- 트랜잭션 충돌이 드물다고 가정하고, 실행 후 검증 단계에서 충돌을 확인하는 동시성 제어 기법
- Lock 없이 실행하고 Commit 전 Validation으로 직렬 가능성 검증

## 키워드
* 3단계: Read Phase → Validation Phase → Write Phase
* 낙관적 가정: 충돌이 드물다고 가정
* Validation: Commit 전 충돌 검사
* Backward/Forward Validation: 검증 방향

## 암기법
* 3단계: "읽검쓰" (Read, Validation, Write)
* 낙관적: "일단 하고 나중에 검사" (Lock 없이 실행 후 검증)
* 적용: "읽기 많고 충돌 적을 때" (Read-intensive workload)

## 구성요소/특징/유형
### 3단계 수행 절차
| 단계 | 영문 | 동작 | 특징 |
|------|------|------|------|
| 1단계 | Read Phase | 읽기/쓰기 연산 수행 | 로컬 복사본에 쓰기, DB 미반영 |
| 2단계 | Validation Phase | 충돌 검사 | 직렬 가능성 검증 |
| 3단계 | Write Phase | DB에 반영 | 검증 통과 시에만 수행 |

### Validation 규칙
| 조건 | Ti가 Tj보다 먼저 시작 (TS(Ti) < TS(Tj)) |
|------|----------------------------------------|
| 규칙 1 | Ti의 Write Phase가 Tj의 Read Phase 전 완료 |
| 규칙 2 | Ti의 Write Set ∩ Tj의 Read Set = ∅ (겹치지 않음) |
| 규칙 3 | Ti의 Write Set ∩ Tj의 Write Set = ∅ |

### Validation 방향
| 방향 | 설명 |
|------|------|
| Backward Validation | 이미 완료된 트랜잭션과 비교 |
| Forward Validation | 현재 진행 중인 트랜잭션과 비교 |

### 장단점
| 구분 | 내용 |
|------|------|
| 장점 | Lock 오버헤드 없음, Deadlock 없음, 읽기 성능 우수 |
| 단점 | 충돌 시 Abort 후 재시작, 쓰기 많으면 비효율 |
| 적합 | 읽기 위주, 충돌 적은 환경 |
| 부적합 | 쓰기 위주, 충돌 빈번한 환경 |

### 동시성 제어 기법 비교
| 기법 | 충돌 처리 | Deadlock | 적합 환경 |
|------|-----------|----------|-----------|
| 2PL (비관적) | 사전 차단 (Lock) | 발생 가능 | 쓰기 위주 |
| Timestamp | 즉시 Abort | 없음 | 균형 |
| 낙관적 | 사후 검증 | 없음 | 읽기 위주 |
